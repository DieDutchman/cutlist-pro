## What Improved This Pass

### H4: Consistency and Standards — P1 issues fully closed
Both `saveAddonPreset()` and `promptAddEdge()` now use `inputPrompt()` instead of native `prompt()`. Every user-facing text-input flow in the app now uses the styled, keyboard-accessible, Escape-dismissable modal overlay. The two-step native `prompt()` chain in `promptAddEdge()` (especially disorienting because the second prompt appeared only after the first was accepted) is replaced with two sequential `inputPrompt()` awaits. This fully resolves the P1 consistency issue flagged in the previous pass.

### H8: Aesthetic and Minimalist Design — P2 border-left violation closed
The builder shelf-crossing warning (line 22854) now uses `border-top:3px solid #c07020`. This was the last remaining `border-left:3px` accent-stripe violation. The interface is now clean of the side-stripe pattern at every surface and severity level.

### H1: Visibility of System Status — P2 animation gap closed
`@keyframes regenPulse` is now guarded by `@media (prefers-reduced-motion: reduce) { #cutlist-regen-pill { animation: none; } }` at lines 2985–2987. Users with OS-level motion reduction no longer see the persistent box-shadow pulse. WCAG 2.3.3 spirit now honoured.

## Priority Issues

**[P2] `inputPrompt` confirm button always reads "RENAME"**
`okBtn.textContent` is hardcoded `'RENAME'` at line 4937 regardless of call context. When `saveAddonPreset()` calls `inputPrompt('Name this preset:', ...)`, the confirm button reads "RENAME". When `promptAddEdge()` calls it for edging size and price, the button reads "RENAME". Semantically wrong; undermines the trust that styled modals are context-aware.
**Fix:** Add an optional `confirmLabel` parameter to `inputPrompt(message, defaultValue, confirmLabel = 'OK')` and pass `'SAVE'` from `saveAddonPreset` and `'ADD'` from `promptAddEdge`.

**[P2] `promptAddEdge()` two-step flow has no back path**
Sequential `inputPrompt()` calls are an improvement over two sequential native `prompt()` calls, but they still leave the user in a broken state: accept the size step, then cancel the price step — the function returns silently, no edge is created, and there is no way back to edit the size already entered. The user must restart from scratch.
**Fix:** Wrap both fields in a single two-field inline modal (or pass both field configs to a `multiInputPrompt()` helper) so back/cancel at step 2 returns to step 1 rather than aborting the whole flow.

**[P2] L1/L2/W1/W2 edge codes not explained at edging dropdowns in unit parts editor**
The ℹ popover exists at the part-table column headers in the Panels, Cutlist, and Quote surfaces. In the unit parts editor (where Jordan adds or edits a part line), the four edging dropdowns are labelled `L1`, `L2`, `W1`, `W2` with no ℹ popover and no tooltip. Jordan on her third session will have learned the codes; Jordan on her first will not.
**Fix:** Add a small `ℹ` icon next to the L1/L2/W1/W2 labels in the unit parts editor that triggers `showEdgeCodes()`, or add a `title` tooltip to each dropdown with a one-line explanation ("L1: Long edge 1 — height dimension").

**[P3] `importState.label` resets correctly on Quote-tab activation (line 7259) and on import clear (line 21912), but not on Import-tab activation**
If a user switches to the Import tab directly (not via Quote), then navigates away and back without clearing, `importState.label` retains the previous file name. Low confusion risk since the label field is editable and visible, but inconsistent with the reset behaviour on Quote activation.
**Fix:** Reset `importState.label = ''` in the `section === '__import__'` branch of `switchTab()`, or on `renderImportView()` when `importState.status === 'idle'`.

## Minor Observations

- `okBtn.textContent` hardcoded `'RENAME'` (line 4937) — exposed by new call sites in `saveAddonPreset` and `promptAddEdge`.
- `promptAddEdge()` two-step sequential `inputPrompt()` is an improvement over native `prompt()` chain, but multi-field case warrants a proper two-field modal.
- `aria-controls` not wired between tab buttons and panel IDs in main nav or Users modal — `role="tab"` and `aria-selected` are present but the panel association is missing for screen readers.
- Return focus on modal close not implemented — `confirmAction()`, `inputPrompt()`, and Users modal do not return focus to the triggering element on close. Minor a11y gap for keyboard-only users.
- `importState.label` resets on Quote-tab activation and import clear but not on Import-tab revisit without clearing.
